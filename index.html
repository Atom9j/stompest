stomp, stomper, stompest!¶

stompest is a full-featured implementation of the STOMP protocol (versions 1.0 and 1.1) for Python 2.6 (and higher) including both synchronous and Twisted clients:

  • The synchronous Stomp client is dead simple. It does not assume anything about your concurrency model (thread vs process) or force you to use it any particular way. It gets out of your way and lets you do what you want.
  • The asynchronous Stomp client is based on Twisted, a very mature and powerful asynchronous programming framework. It supports destination specific message and error handlers (with default “poison pill” error handling), concurrent message processing, graceful shutdown, and connect, receipt, and disconnect timeouts.

Both clients make use of a generic set of components in the protocol module each of which can be used independently to roll your own STOMP client:

  • a wire-level STOMP frame parser StompParser and compiler StompFrame,
  • a faithful implementation of the syntax of the STOMP 1.0 and 1.1 protocols with a simple stateless function API in the protocol.commands module,
  • a generic implementation of the STOMP 1.0 and 1.1 session state semantics in StompSession, such as protocol version negotiation at connect time, transaction and subscription handling (including a generic subscription replay scheme which may be used to reconstruct the session’s subscription state after a forced disconnect),
  • and StompFailoverTransport, a failover transport URI scheme akin to the one used in ActiveMQ.

This module is thoroughly unit tested and (in version 1.x) production hardened for the functionality used by the current maintainer and Mozes — persistent queueing on ActiveMQ. The module is tested with Python 2.6 and 2.7, Twisted 11.0 and 12.0 (it should work with Twisted 10.1 and higher), and ActiveMQ 5.5.1, 5.6, and 5.7. The integration tests also pass against RabbitMQ 2.8.7 and Apollo 1.5. All of these brokers were tested with STOMP protocols 1.0 and 1.1. Minor enhancements may be required to use this STOMP adapter with other brokers.

A few words about history: When stompest was brought into being, there were already several STOMP related modules around, but none of them did address the main requirements: a simple access to the STOMP protocol which stays out of your way (the synchronous Stomp client) features and an event-driven implementation which leverages the power of the Twisted framework (the asynchronous Stomp client). The stomper package provided an abstract implementation of the STOMP protocol which stompest 1 took advantage of: stomp, stomper, stompest! During the redesign which lead to stompest 2, the stomper package turned out to suffer from being out-of-maintenance, so the appealing feature of an abstract STOMP API was insourced.
